# Интеграция с Яндекс.Картами — отзывы и рейтинг организации

Тестовое задание: приложение из двух экранов (авторизация + настройки/отзывы),
которое по ссылке на карточку организации в Яндекс.Картах вытягивает её отзывы,
средний рейтинг и счётчики.

**Стек:** Laravel 13 (API) · Vue 3 + Composition API (SPA) · Python + Playwright (парсер) · SQLite/MySQL.

---

## Архитектура

```
┌─────────────┐   HTTP    ┌──────────────┐   HTTP    ┌─────────────────────┐
│  Vue 3 SPA  │ ───────►  │  Laravel API │ ───────►  │ Python + Playwright  │ ──► Яндекс.Карты
│ (Vite :5173)│  /api/... │   (:8000)    │  /parse   │      (:8001)         │
└─────────────┘           └──────┬───────┘           └─────────────────────┘
                                 │
                                 ▼
                          ┌─────────────┐
                          │ SQLite/MySQL │  (пользователь, организация, отзывы)
                          └─────────────┘
```

**Почему три отдельных сервиса:**

- **Laravel** — бизнес-логика, авторизация, хранение, отдача данных фронту с пагинацией.
- **Python + Playwright (отдельный сервис)** — парсинг. PHP плохо управляет headless-браузером,
  а Playwright на Python — индустриальный стандарт для обхода динамической подгрузки и защиты от ботов.
  Laravel общается с ним по простому HTTP (`POST /parse`), поэтому парсер можно масштабировать
  или заменить независимо.
- **Vue 3 SPA** — интерфейс, состояния загрузки/ошибок видны пользователю.

---

## Подход к парсингу (центральная часть задания)

У Яндекс.Карт **нет официального API**, отзывы подгружаются JavaScript'ом по мере прокрутки,
плюс есть защита от ботов. Поэтому:

1. **Headless-браузер (Playwright + Chromium)**, а не `requests`/`BeautifulSoup` — потому что
   нужен реальный движок, исполняющий JS. Обычный HTTP-запрос вернёт пустой каркас страницы.

2. **Маскировка под человека** для обхода антибота:
   - флаг `--disable-blink-features=AutomationControlled` убирает признак автоматизации;
   - реальный `User-Agent`, `viewport`, `locale: ru-RU`;
   - паузы между прокрутками (имитация чтения).

3. **Сбор всех отзывов (~600)** — прокручиваем контейнер списка вниз порциями, после каждой
   ждём подгрузки и собираем новые отзывы. Дедупликация по автору. **Стоп-условие:** три
   прокрутки подряд без новых отзывов (одна может не успеть подгрузиться при медленной сети —
   поэтому именно три, а не одна).

4. **Статистика** (средний рейтинг, число оценок, число отзывов) читается из шапки карточки
   регулярными выражениями (строка вида «1 234 оценки · 567 отзывов»).

### Решение по пагинации и кэшированию (тоже оценивается)

Мы **не** дёргаем Яндекс при каждом открытии страницы. Логика такая:

- При сохранении ссылки парсер **один раз** вытягивает все ~600 отзывов → Laravel кладёт их в БД.
- Постраничная навигация (50 на страницу) отдаётся **из БД** через `paginate(50)` — мгновенно,
  без нагрузки на Яндекс и без риска поймать капчу на каждом клике.

Это компромисс: первый парсинг идёт 1–2 минуты, зато дальше всё быстро и устойчиво. Альтернатива
(тянуть с Яндекса по 50 на каждый запрос) дала бы свежесть данных ценой скорости и риска бана —
для сервиса отзывов кэш оправдан, актуализацию можно делать по кнопке/крону.

### Обработка ошибок и крайних случаев

- невалидная ссылка → `422` с понятным сообщением (валидация в `StoreOrganizationRequest`);
- парсер недоступен / упал → `502`, статус организации `error`, текст ошибки виден в UI;
- пустой ответ / изменилась разметка → фиксируется как ошибка, отзывы не затираются мусором
  (запись в БД идёт транзакцией).

## Структура БД (миграции)

- **users** — стандартная таблица Laravel (один сид-пользователь).
- **organizations** — карточка пользователя: `url`, `yandex_id`, `name`, кэш статистики
  (`average_rating`, `ratings_count`, `reviews_count`), `parse_status` / `parse_error` / `parsed_at`.
- **reviews** — отзывы: `organization_id`, `author`, `review_date`, `text`, `rating`.

## Запуск локально

Требуется PHP 8.2+, Composer, Node 18+, Python 3.11+ (для парсера).

### 1. Бэкенд (Laravel)

```bash
cd backend
cp .env.example .env            # при необходимости
composer install
php artisan key:generate
touch database/database.sqlite  # SQLite-файл
php artisan migrate --seed      # таблицы + сид-пользователь
php artisan serve --port=8000
```

Сид-пользователь: **admin@example.com** / **password**

### 2. Парсер (Python)

```bash
cd parser
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt
playwright install chromium
uvicorn main:app --port 8001
```

### 3. Фронтенд (Vue)

```bash
cd frontend
npm install
npm run dev                     # http://localhost:5173
```

Vite проксирует `/api` на Laravel (`:8000`). Laravel ходит в парсер по `PARSER_URL`.

### Через docker-compose

```bash
docker-compose up --build
```

Поднимет всё разом: nginx + Laravel + MySQL + Vue + парсер.

## Переменные окружения

| Переменная      | Где      | Назначение                                          |
|-----------------|----------|-----------------------------------------------------|
| `DB_CONNECTION` | backend  | `sqlite` (локально) или `mysql` (Docker)            |
| `PARSER_URL`    | backend  | Адрес Python-парсера, напр. `http://localhost:8001` |
| `VITE_API_URL`  | frontend | Базовый URL API (по умолчанию `/api` через прокси)  |

## Что доделал бы, имея больше времени

- **Парсинг через очередь**: сейчас парсинг синхронный (запрос ждёт 1–2 мин). Вынес бы
  в очередь (Laravel Queue + Job), фронт опрашивал бы статус — лучше UX и нет таймаутов.
- **Устойчивость к смене разметки**: селекторы Яндекса меняются. Добавил бы разбор внутренних
  JSON-запросов карточки (а не только DOM) и набор резервных селекторов.
- **Капча/прокси**: ротация прокси и user-agent'ов, обработка капчи для стабильного парсинга объёма.
- **Тесты**: feature-тесты на API (логин, валидация, пагинация) и мок-тест сервиса парсера.
- **Инкрементальное обновление** отзывов вместо полной перезаписи, кнопка «обновить».
